自动数据绑定

在节点及节点属性上使用数据引用时能够自动将该数据源与该节点进行数据绑定操作，当数据源上的数据发生变更时将同时影响该节点

提问环节：

数据绑定需要手动设定吗？

不需要。所有使用数据引用的节点都能自动建立绑定关系，并且在逻辑性节点中也能自动识别绑定对象

计时：{{time + startTime}} 秒

当数据源 time 或 startTime 任意一个数据源属性发生改变时都会触发节点修改

可编程的 CSS

可编程 CSS 语言，支持继承和类等，规范书写，无需书写适配前缀

@class coverBackground (color) { background-color: [color]; background-position: center; background-repeat: no-repeat; background-size: cover; } @if (device.os.ios) { .demo { top: [header-height]; height: calc(100vh + 12dp - 1px); @class: coverBackground(#fff); } }

提问环节：

项目还需要 SASS、LESS、Stylus 等吗？

答案肯定是不需要。IOING 是一个纯前端引擎，所有服务都是前端运行的结果


运行在前端的 CSS 引擎会不会很慢？

当然不会。IOING 的 CSS 引擎 是一个非常高效的解释器，一般整个模块的 CSS 文件的编译速度都在 0.002s 内完成


为什么采用前端编译？

前端可编程 CSS 的特点是能根据不同设备环境以及模版数据灵活编译，具有针对性和实效性，减小文件传输提及的同时更减少了日常维护成本

提问环节：

模版数据变更时是如何进行局部更新的？

引擎在编译输出前就已经把每个节点及其节点上的引用数据建立了点对点的绑定关系，当数据变更时引擎能立刻针对关联节点进行更新操作，这就像是一个量子态，数据发生改变同时也会体现在 DOM 上

shadowroot 隔离

使用shadow 标签包裹的子节点被封装为 shadow dom，使其脱离外部 DOM Tree 节点，被包裹元素的 id、class、 CSS 样式、js 变量 都能够和父级解耦

blue red

提问环节：

浏览器不支持 shadow dom 特性怎么办？

引擎对不支持该特性的浏览器能够自动采用无感知降级方案，开发者无需担心两者间的区别。另外该特性其实也是组件的基本特性，从而能够保证组件内外的逻辑互不干扰

沙盒化

HTML 的设计之初是为了传递和共享信息，因而它已开始更偏向于构建内容而非构建应用。当文档要传递的内容过多时则无法通过一个文档来显示所有内容，我们需要使用多个文档协作展示，即通过链接跳转的方式展示更多内容，但这种方式会导致文档脱离了原有的历史关系，同时在你的窗口运行环境中的数据也将一并随之丢失，后来我们开始流行使用 Ajax 技术在同一个页面中展示更多数据

document-A

document-B

document-B

Ajax Ajax

但 Ajax 也同样带来了诸多问题，载入的大量内容节点都堆积在同一个 DOM Tree 下，每一次节点更改都造成大量性能损耗，以至于性能也随之下降

IOING 模块支持三种运行模式为sandbox和shadowbox及传统模式，在沙盒模式下模块拥有独立的作用域空间以保证模块间互不干扰，当模块被卸载时也能够自动释放占用内存，并解除模块产生的僵尸事件
